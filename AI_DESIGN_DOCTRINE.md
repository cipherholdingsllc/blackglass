# AI Design Doctrine

AI-assisted teams need more than prompts. They need durable design doctrine that can be reused, inspected, and challenged.

## Rules

- Treat design docs as architecture.
- Use tokens as constraints, not decoration.
- Give each pattern a job before giving it polish.
- Review screenshots, not just code.
- Name exhibits like durable artifacts, not demos.
- Avoid one-off styling unless it becomes a documented pattern.
- Reject visual ideas that cannot explain state, hierarchy, focus, trust, or sequence.

## Why Prompts Are Not Enough

Prompts can produce quick variation. They do not automatically preserve taste across many screens. Blackglass turns taste into reusable rules so a build pass has something stronger to follow than vibes.

## Review Loop

1. Define the interface job.
2. Choose the semantic state.
3. Apply tokens.
4. Render the screenshot.
5. Reject generic outputs.
6. Keep the smallest version that still feels distinctive.

## Anti-Drift Checks

- Does every glow mean something?
- Does every glass layer clarify depth or focus?
- Can color be understood without relying on color alone?
- Would the surface still work if motion were reduced?
- Is the screenshot memorable in five seconds?
